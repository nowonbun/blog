데이터베이스 설계

 이번에는 가계부의 회원관리 데이터베이스 설계하겠습니다. 회원 관리 부분은 많은 프로그램이 기본으로 설계되는 부분이기도 합니다. 최근에는 OAuth를 통해 회원관리를 최소화하는 프로그램들도 간혹 있기는 하지만 oAuth를 사용한다 하더라도 회원 정보는 생략해도 회원 그룹관리, 권환관리는 필요한 부분입니다. 회원관리를 입력할 때 너무 많은 정보를 수집하게 되면 개인정보 보호법에 따라 수집의 절차가 있고 예를 들면 동의 절차를 받아야 하고, 보관의 기한이 있으며, 폐기를 할 때는 어떻게 해야 하는지, 어느 부분은 암호화를 해야 하는지등의 절차가 되어있습니다. 이를 지키지 않으면 벌금 내거나 동의 없이 제 3자에게 제공을 하면 징역을 받고 사이트 강제 폐쇄까지 해야하는 상황에 오게 됩니다.
 
 제가 설명은 거창하게 했지만, 실상은 아이디 패스워드, 이름을 대신할 수 있는 별명 정도만 수집받으면 개인정보에 걸리는 영역은 없습니다. 이번 가계부 같은 경우는 우리 가족만 쓰게 될 목표가 있어서 특히 회원가입은 더더욱 필요 없고 정보 자체도 Google oAuth를 통해 오는 id, email, 별명, 프로필 이미지 정도만 수집할 생각입니다.

 마스터는 총 3개로 그룹, View권한, Action권한입니다. 보통의 프로그램들은 그룹 테이블을 마스터 테이블이 아니고 트랜잭션으로 두어 사용자가 그룹관리도 가능하게끔 설계를 하는 게 보통입니다. 예를 들면 기본 유저는 Guest를 해서 사용 유저가 그룹을 추가하여 그 그룹으로 이동하거나 등등입니다. 그러나 그렇게 되면 그룹관리를 해야하는 영역도 생각해야 하는데 집에서 쓰는 가계부는 일단 사용 유저가 저 포함 두 명, 세 명으로 한정되어 있기 때문에 굳이 그룹관리를 만들어서 프로그램을 복잡하게 할 필요가 있을까 싶어서 마스터로 정의해 버렸습니다.
 우리 가족이 아닌 외부에서 접속을 하게 되면 Guest, 일반 사용 유저는 User, 저는 Admin으로 슈퍼 유저 그룹의 단 3그룹으로 생각하고 있습니다.
 
 View권한과 Action권한은 권한에 대한 정의입니다. 앞서 이야기했듯이 외부 유저가 로그인을 하면 Guest권한이 디폴트로 생성할 생각입니다. 그러나 가계부는 우리 집의 기밀이기 때문에 이게 불특정 다수에게 공개된다면 조금 창피한 일이 발생하겠네요. 그래서 Guest그룹의 경우는 권한이 없게끔 하고 User의 경우는 일단 가계부를 작성하게는 하나 카테고리 추가 삭제 등을 제한을 둘 생각입니다. Admin은 모든 권한을 넣을 생각입니다.
 
그리고 User 테이블은 트랜잭션이긴 하지만 기본 키가 idx(int형식의 auto_increment 제약키)형식이 아니고 id자체가 고유키이기 때문에 id로 키를 설정했습니다. 사실 email까지 같이 이중키를 만들까도 했는데 괜히 테이블구조만 복잡해 질 거 같아서 ID만 넣었습니다. 

마지막은 그룹의 Map입니다. 그룹 맵이란 사용자와 그룹관계를 n:n으로 설정하는 것입니다. 사실 이게 이렇게까지는 필요가 없습니다. 하나의 유저가 여러 개의 그룹을 속하는 게 오히려 관리 면에서 더 복잡해지기 때문이고 우리 시스템은 사실 그렇게까지는 필요 없습니다. 
그러나 제가 요번에는 가계부에서 끝나는 게 아니고 앞으로 우리집 스케줄 공유와 게시판 등등으로 시스템을 키워나갈 생각입니다. 혹시 나중에 그룹관리가 어떻게 될지 모르니 혹시나 하는 형태로 작성해 넣었습니다.
CREATE TABLE MAP_VIEW_ROLE_GROUP(
  VIEWCODE CHAR(4) NOT NULL,
  GROUPCODE CHAR(1) NOT NULL,
  
  PRIMARY KEY (VIEWCODE, GROUPCODE),
  FOREIGN KEY (VIEWCODE) REFERENCES MST_VIEWROLE(CODE),
  FOREIGN KEY (GROUPCODE) REFERENCES MST_GROUP(CODE)
);

CREATE TABLE MAP_VIEW_ROLE_USER(
  VIEWCODE CHAR(4) NOT NULL,
  ID VARCHAR(100) NOT NULL,
  
  PRIMARY KEY (VIEWCODE, ID),
  FOREIGN KEY (VIEWCODE) REFERENCES MST_VIEWROLE(CODE),
  FOREIGN KEY (ID) REFERENCES TSN_USER(ID)
);

CREATE TABLE MAP_ACTION_ROLE_GROUP(
  ROLECODE CHAR(4) NOT NULL,
  GROUPCODE CHAR(1) NOT NULL,
  
  PRIMARY KEY (ROLECODE, GROUPCODE),
  FOREIGN KEY (ROLECODE) REFERENCES MST_ACTIONROLE(CODE),
  FOREIGN KEY (GROUPCODE) REFERENCES MST_GROUP(CODE)
);

CREATE TABLE MAP_ACTION_ROLE_USER(
  ROLECODE CHAR(4) NOT NULL,
  ID VARCHAR(100) NOT NULL,
  
  PRIMARY KEY (ROLECODE, ID),
  FOREIGN KEY (ROLECODE) REFERENCES MST_ACTIONROLE(CODE),
  FOREIGN KEY (ID) REFERENCES TSN_USER(ID)
);
이번에는 권환테이블과 유저 또는 그룹의 권한 설정입니다. 이 역시 n:n의 형태이기 때문에 map으로 작성했습니다. 제가 생각하는 권한을 이렇습니다.
먼저 "가"라는 인물이 "A"라는 그룹에 속해있으면 "Add"의 권한을 A그룹은 권한이 없지만 "가"라는 특정인물이 권한이 있으면 "Add"를 사용할 수 있게 합니다.
반대로 A그룹이 "Add"의 권한을 가지고 있으면 "가"라는 인물이 권한이 있던 없던 간에 사용할 수 있습니다.
"Add"권한은 하나의 그룹만 사용하는 권한이 아니기 때문에 n:n의 조건이 성립합니다.
