[Java] JPAでトランザクションの使用方法とオブサーバーパターンで共通トランザクション関数を作り方

Study / Java    作成日付 : 2019/10/10 07:29:43   修正日付 : 2020/01/17 07:46:38

こんにちは。明月です。


データベースでデータの無欠性を保持するためにはオブサーバーパターンでトランザクションを使うべきです。トランザクションを利用して無欠性(データベースの中で重複データ、整合性が合わないデータがない状況)を保持するという意味ではエラーが発生する時にコミット、ロールバックが適切的にできるということですね。

例えば、「User」テーブルの派生で「Info」テーブルがあります。

「Info」のカラムでレファレンスが「User」テーブルと紐付いて「Not null」の条件になっています。

そうすると「Info」データにデータを入れるようには先に「User」テーブルにもデータをインサートあるいは既存でデータが必要です。それで先に「User」テーブルにデータをインサートして「Info」テーブルにインサートしようと実装しましたが、プログラムで問題が発生してインサートができなくなりました。

そうなら先にインサートした「User」テーブルのデータをロールバックしなければならないです。


ただ、私は２つのテーブルで考えたんですが、もしテーブルレファレンスがすごく複雑だし様々で紐付いたと思えばエラーが発生する時にデータを全てロールバックしなければならないです。

private static void run() throws Exception {
  EntityManagerFactory emf = Persistence.createEntityManagerFactory("JpaExample");
  EntityManager em = emf.createEntityManager();
  // Userデータを生成
  User user = new User();
  user.setId("nowonbun");
  user.setName("明月");
  user.setInfos(new LinkedList<>());
 
  em.persist(user);
 
  // Infoデータを生成
  Info info = new Info();
  user.addInfo(info);
  info.setAge(20);
  info.setInfo2s(new LinkedList<>());
 
  em.persist(info);
  if(true) {
    // ここでエラーが発生した。上の「User」データと「Info」データをロールバックしなければならない。
    throw new Exception();  
  }
  // Info2データを生成
  Info2 info2 = new Info2();
  info.addInfo2(info2);
  info2.setBirth(new Date());
  em.persist(info);
}

実は上のソースは実行もできないです。ただ、例のために実装したことです。

上の例をみれば私が「User」テーブルにインサートして「Info」テーブルにインサートしました。そしてエラーが発生します。

エラーが発生すればこのデータはインサートしなかったので、ロールバックをしなければならないです。

そうすれば、上の状況になると「User」テーブルと「Info」テーブルにデータを探して削除しなければならないです。ただ、生成したことは削除で終わりますが、更新したとは解決が難しくなります。

それでトランザクションが必要です。

public class Main {
  private static void run() throws Exception {
    EntityManagerFactory emf = Persistence.createEntityManagerFactory("JpaExample");
    EntityManager em = emf.createEntityManager();
    // トランザクションを持ち込む。
    EntityTransaction transaction = em.getTransaction();
    // トランザクションを開始
    transaction.begin();
    try {
      // Userデータを生成
      User user = new User();
      user.setId("TEST");
      user.setName("TEST");
      user.setInfos(new LinkedList<>());
      em.persist(user);
 
      // Infoデータを生成
      Info info = new Info();
      user.addInfo(info);
      info.setAge(20);
      info.setInfo2s(new LinkedList<>());
      em.persist(info);
 
      if (true) {
        throw new Exception();
      }
 
      // Info2データを生成
      Info2 info2 = new Info2();
      info.addInfo2(info2);
      info2.setBirth(new Date());
      //予想する結果はUserテーブルとInfoテーブル、Info2テーブルにインサート
      em.persist(user);
      // transactionをコミットする。
      transaction.commit();
    } catch (Throwable e) {
      // errorが発生するとトランザクションをロールバックする。
      if (transaction.isActive()) {
        transaction.rollback();
      }
      e.printStackTrace();
    }
  }
 
  public static void main(String... args) {}
    try {
      run();
    } catch (Throwable e) {
      e.printStackTrace();
    }
  }
}

上のソースをみれば「EntityManager」から「Transaction」を持ち来ます。そして「transaction.begin()」でトランザクションを開始すると「commit」になる前にデータを一時的に持っていって「commit」になると一括でデータを処理します。途中でエラーが発生して「rollback」関数が呼ばれたらデータを変わらないです。

そのため、トランザクションを使わなければならない理由になります。参考は「Select」の検索だけにはトランザクションが必要ないです。


トランザクションの意味は分かりました。

そうすれば、上のソースのスタイルをオブサーバーパターンとラムダ式を利用してもっと見やすく作りましょう。

link - [Java スタディ - 20] 匿名クラス、そしてラムダ式(Lambda expression)

public class Main {
  EntityManagerFactory emf = Persistence.createEntityManagerFactory("JpaExample");
  EntityManager em = emf.createEntityManager();
 
  interface EntityManagerCallable<V> {
    V run(EntityManager em);
  }
 
  interface EntityManagerRunable {
    void run(EntityManager em);
  }
 
  //リターン値があるトランザクション
  public <V> V transaction(EntityManagerCallable<V> callable) {
    return transaction(callable, false);
  }
 
  public <V> V transaction(EntityManagerCallable<V> callable, boolean readonly) {
    EntityManager em = emf.createEntityManager();
    EntityTransaction transaction = em.getTransaction();
    // トランザクションを開始する。
    transaction.begin();
    try {
      // ラムダ式を実行する。
      V ret = callable.run(em);
      // readonlyが「true」なら「rollback」する。
      if (readonly) {
        transaction.rollback();
      } else {
        // トランザクションをコミットする。
        transaction.commit();
      }
      return ret;
    } catch (Throwable e) {
      // エラーが発生すればロールバックする。
      if (transaction.isActive()) {
        transaction.rollback();
      }
      throw new RuntimeException(e);
    } finally {
      em.clear();
      em.close();
    }
  }
  
  // リターン値がないトランザクション
  public void transaction(EntityManagerRunable runnable) {
    transaction(runnable, false);
  }
 
  public void transaction(EntityManagerRunable runnable, boolean readonly) {
    EntityManager em = emf.createEntityManager();
    EntityTransaction transaction = em.getTransaction();
    // トランザクションを開始
    transaction.begin();
    try {
      // ラムダ式を実行する。
      runnable.run(em);
      // readonlyが「true」ならロールバックする。
      if (readonly) {
        transaction.rollback();
      } else {
        // トランザクションをコミットする。
        transaction.commit();
      }
    } catch (Throwable e) {
      // エラーが発生すればロールバックする。
      if (transaction.isActive()) {
        transaction.rollback();
      }
      throw new RuntimeException(e);
    } finally {
      em.clear();
      em.close();
    }
  }
 
  private void run() {
    // もしエラーが発生すればロールバックするし、なければコミットする。
    transaction((em) -> {
      //Userデータを生成
      User user = new User();
      user.setId("TEST");
      user.setName("TEST");
      user.setInfos(new LinkedList<>());
      em.persist(user);
 
      //Infoデータを生成
      Info info = new Info();
      user.addInfo(info);
      info.setAge(20);
      info.setInfo2s(new LinkedList<>());
      em.persist(info);
 
      //Info2データを生成
      Info2 info2 = new Info2();
      info.addInfo2(info2);
      info2.setBirth(new Date());
      //予想する結果はUserテーブルとInfoテーブル、Info2テーブルにInsertする
      em.persist(user);
    });
  }
 
  public static void main(String... args) {
    Main main = new Main();
    main.run();
  }
}

上のソースにはトランザクションをラムダ式で結び込みました。

それでデータベースを接続してデータを入れる時には共通トランザクション関数を使用するとエラーがある場合はロールバックをするし、なければコミットします。

上のパターンがオブサーバーパターンですが、デザインパターンを使うと上のソースらしく、可読性が上がりますね。
